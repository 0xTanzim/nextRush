NextRush is built on three principles that shape every design decision.

## Design Philosophy

### 1. Minimal Core

The core framework is under 3,000 lines of code. Everything else is modular.

This matters because:

- Fewer lines means fewer bugs
- Small core is easier to audit for security
- Predictable behavior — no hidden surprises

### 2. Dual Paradigm

NextRush supports both **functional** and **class-based** programming equally well.

<Tabs items={['Functional', 'Class-based']}>
  <Tab value="Functional">
    ```ts
    import { createApp } from '@nextrush/core';
    import { createRouter } from '@nextrush/router';

    const app = createApp();
    const router = createRouter();

    router.get('/users', (ctx) => {
      ctx.json([{ id: 1, name: 'Alice' }]);
    });

    app.route('/', router);
    ```

  </Tab>
  <Tab value="Class-based">
    ```ts
    import 'reflect-metadata';
    import { Controller, Get } from '@nextrush/decorators';
    import { Service } from '@nextrush/di';

    @Service()
    class UserService {
      findAll() {
        return [{ id: 1, name: 'Alice' }];
      }
    }

    @Controller('/users')
    class UserController {
      constructor(private users: UserService) {}

      @Get()
      findAll() {
        return this.users.findAll();
      }
    }
    ```

  </Tab>
</Tabs>

Both styles produce identical runtime behavior. Choose based on your team's preference.

### 3. No Magic

Every automatic behavior is documented and has an escape hatch.

When NextRush does something automatically (like JSON serialization or parameter injection), the documentation explains:

- What happens
- When it happens
- How to override or opt out

## Core Components

NextRush has five core components that work together:

```
┌─────────────────────────────────────────────────────────────┐
│                      Your Application                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                     Application                              │
│                                                              │
│  • Manages middleware stack                                  │
│  • Installs plugins                                          │
│  • Creates request handler callback                          │
└─────────────────────────────────────────────────────────────┘
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
     ┌───────────┐   ┌───────────┐   ┌───────────┐
     │ Middleware│   │  Router   │   │  Plugins  │
     │           │   │           │   │           │
     │ Composable│   │ Segment   │   │ Extensible│
     │ pipeline  │   │ trie      │   │ hooks     │

                            │
                            ▼
```

### Application

The `Application` class is your entry point. It has four responsibilities:

1. **Register middleware** — `app.use(middleware)`
2. **Mount routers** — `app.route('/api', router)`
3. **Install plugins** — `app.plugin(myPlugin)`
4. **Create handler** — `app.callback()` returns the request handler

```ts
import { createApp } from '@nextrush/core';

const app = createApp();

// 1. Register middleware
app.use(cors());
app.use(helmet());

// 2. Install plugins
app.plugin(loggerPlugin);

// 3. Handler is used internally by adapters
const handler = app.callback();
```

### Context

Every request creates a fresh `Context` object. This is your interface to the request and response.

```ts
// INPUT — Read request data
ctx.method; // 'GET', 'POST', etc.
ctx.path; // '/users/123'
ctx.params; // { id: '123' }
ctx.query; // { page: '1' }
ctx.body; // Parsed request body
ctx.headers; // Request headers
ctx.ip; // Client IP address

// OUTPUT — Send response
ctx.status = 201;
ctx.json(data); // Send JSON
ctx.html(str); // Send HTML
ctx.send(buffer); // Send raw data
ctx.redirect(url);

// MIDDLEWARE
await ctx.next(); // Call next middleware
```

<Callout type="info">
  The Context interface is the same across all runtimes. Your code works on Node.js, Bun, Deno, and
  Edge without changes.
</Callout>

### Middleware

Middleware functions transform requests and responses. They execute in order, wrapping around your handlers.

```ts
app.use(async (ctx) => {
  const start = Date.now();

  await ctx.next(); // Call next middleware

  const ms = Date.now() - start;
  console.log(`${ctx.method} ${ctx.path} - ${ms}ms`);
});
```

Middleware runs in an **onion model**:

```
Request  ─►  [Middleware 1]  ─►  [Middleware 2]  ─►  [Handler]
                   │                   │                │
Response ◄──  [Middleware 1]  ◄──  [Middleware 2]  ◄───┘
```

Each middleware can:

- Run code before the next middleware
- Call `await ctx.next()` to continue
- Run code after the next middleware returns
- Short-circuit by not calling `next()`

### Router

The router matches URLs to handlers using a segment trie for O(d) lookup where d is the number of path segments.

```ts
import { createRouter } from '@nextrush/router';

const router = createRouter();

router.get('/users', listUsers);
router.get('/users/:id', getUser);
router.post('/users', createUser);
router.put('/users/:id', updateUser);
router.delete('/users/:id', deleteUser);

// Mount router on app
app.route('/api', router);
```

Route parameters are extracted automatically:

```ts
router.get('/users/:id/posts/:postId', (ctx) => {
  const { id, postId } = ctx.params;
  // id = '123', postId = '456' for /users/123/posts/456
});
```

### Adapters

Adapters bridge NextRush to specific runtimes. They convert platform-specific requests into the universal `Context` interface.
