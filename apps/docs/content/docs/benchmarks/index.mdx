This page provides transparent, reproducible benchmarks comparing NextRush v3 to popular Node.js frameworks. All numbers come from automated benchmark runs you can reproduce yourself.

## Why Benchmarks Matter (And Why They Don't)

Performance claims without context are misleading. "35,000 RPS" means nothing without knowing:

- What hardware was used?
- What was the test scenario?
- How does it compare to alternatives?

In production, database queries, external API calls, and business logic dominate response time — not framework overhead. Use these benchmarks to understand relative performance, not absolute numbers.

## Detailed Results by Scenario

### Hello World (Baseline)

A JSON response with no routing or parsing — measures raw framework overhead.

```typescript
app.get('/', (ctx) => {
  ctx.json({ message: 'Hello World' });
});
```

| Framework       | RPS    | Latency (p50) | Latency (p99) |
| --------------- | ------ | ------------- | ------------- |
| Fastify         | 48,045 | 15ms          | 37ms          |
| **NextRush v3** | 43,268 | 16ms          | 43ms          |
| Hono            | 37,476 | 19ms          | 48ms          |
| Koa             | 34,683 | 20ms          | 56ms          |
| Express         | 23,739 | 50ms          | 69ms          |

### Route Parameters

Dynamic routes with path parameters — measures router performance.

```typescript
// GET /users/12345
app.get('/users/:id', (ctx) => {
  ctx.json({ id: ctx.params.id });
});
```

| Framework       | RPS    | Latency (p50) | Latency (p99) |
| --------------- | ------ | ------------- | ------------- |
| Fastify         | 45,852 | 15ms          | 37ms          |
| **NextRush v3** | 38,983 | 19ms          | 45ms          |
| Hono            | 35,144 | 20ms          | 50ms          |
| Koa             | 33,893 | 21ms          | 52ms          |
| Express         | 22,228 | 44ms          | 70ms          |

### Query String Parsing

URL with query parameters — measures query parser performance.

```typescript
// GET /search?q=benchmark&limit=10
app.get('/search', (ctx) => {
  ctx.json(ctx.query);
});
```

| Framework       | RPS    | Latency (p50) | Latency (p99) |
| --------------- | ------ | ------------- | ------------- |
| Fastify         | 36,618 | 20ms          | 51ms          |
| **NextRush v3** | 30,876 | 23ms          | 47ms          |
| Hono            | 28,619 | 27ms          | 50ms          |
| Koa             | 27,637 | 26ms          | 55ms          |
| Express         | 20,769 | 47ms          | 58ms          |

### POST with JSON Body

POST request with JSON parsing — measures body parser performance.

```typescript
app.post('/users', (ctx) => {
  ctx.json({ received: ctx.body });
});
```

| Framework       | RPS    | Latency (p50) | Latency (p99) |
| --------------- | ------ | ------------- | ------------- |
| Fastify         | 21,412 | 45ms          | 71ms          |
| **NextRush v3** | 20,438 | 47ms          | 73ms          |
| Koa             | 17,664 | 55ms          | 75ms          |
| Express         | 14,417 | 66ms          | 104ms         |
| Hono            | 12,625 | 75ms          | 114ms         |

<Callout type="info" title="Body Parser Fairness">
  NextRush, Koa, and Fastify apply body parsing only on POST routes. Express applies
  `express.json()` globally (though it skips non-JSON requests). Hono parses the body on-demand via
  `c.req.json()`. These differences affect POST performance rankings.
</Callout>

### Repeated Baseline (Server Warm)

A second `GET /` run after all other tests have completed — measures performance with a fully warmed server.

| Framework       | RPS    | Latency (p50) | Latency (p99) |
| --------------- | ------ | ------------- | ------------- |
| Fastify         | 48,493 | 15ms          | 32ms          |
| **NextRush v3** | 43,283 | 16ms          | 41ms          |
| Hono            | 38,759 | 18ms          | 39ms          |
| Koa             | 35,566 | 20ms          | 47ms          |
| Express         | 23,783 | 51ms          | 63ms          |

<details>
<summary>Why Fastify is faster</summary>

Fastify achieves higher throughput through:

- **Schema-based JSON serialization** via `fast-json-stringify` — avoids `JSON.stringify` overhead when schemas are defined
- **Ahead-of-time route compilation** — routes are compiled at startup
- **Find-my-way router** — highly optimized radix tree router

These optimizations require upfront schema definitions and add configuration complexity.

</details>

## Why NextRush Doesn't Chase Maximum RPS

NextRush prioritizes developer experience over squeezing every last request per second:

| Priority              | Trade-off                                                 |
| --------------------- | --------------------------------------------------------- |
| **Clean API**         | No schema boilerplate required                            |
| **Modularity**        | Pay only for what you use                                 |
| **Zero dependencies** | No external runtime deps (except reflect-metadata for DI) |
| **Dual paradigm**     | Functional and class-based styles                         |

The ~13% gap from Fastify is often negligible in production where database queries, network I/O, and business logic dominate response time.

## Hardware Impact

Performance varies by hardware. These numbers were measured on a mid-range laptop (i5-8300H). You should expect different absolute numbers on your hardware.

<Callout type="warn" title="Don't Trust Absolute Numbers">
  Always compare frameworks on the **same hardware** with the **same test conditions**. The numbers
  on this page come from a mid-range laptop. Server hardware typically produces 2–4x higher
  throughput.
</Callout>

## Run Your Own Benchmarks

Clone the repository and run:

```bash
cd apps/performance-ultra
pnpm install
pnpm bench           # Full benchmark (30s per test)
pnpm bench --quick   # Quick benchmark (10s per test)
```

Results are saved to `results/comparison/latest/REPORT.md`.

<details>
<summary>Benchmark methodology</summary>

1. **Warm-up**: 3 seconds before each test
2. **Duration**: 10 seconds per test (quick mode) or 30 seconds (default)
3. **Isolation**: Each framework runs in a separate process
4. **Same handlers**: Identical route handlers across frameworks
5. **Single run**: Results are from a single benchmark pass (not averaged across runs)

All server implementations and benchmark scripts are in `apps/performance-ultra/`. Review them to verify fairness.

</details>

## The Honest Take

Instead of asking "which framework is fastest?", ask:

> **"Is this framework fast enough for my use case?"**

For most APIs, all modern frameworks are fast enough. The bottleneck is almost always database queries, external API calls, business logic, or network latency.

NextRush's 35,000+ RPS baseline on a laptop means framework overhead is unlikely to be your bottleneck.
