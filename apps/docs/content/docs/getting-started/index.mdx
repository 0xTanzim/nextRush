# Introduction to NextRush

NextRush is a backend framework for Node.js, Bun, Deno, and Edge runtimes. It's designed for developers who want **speed without sacrificing clarity**.

## The Problem with Existing Frameworks

If you've built backend APIs before, you've likely encountered these frustrations:

**Express** — Fast to start, but lacks structure. TypeScript is bolted on. Middleware composition is manual.

**Koa** — Clean async/await, but the ecosystem is fragmented. Many essential features require third-party packages with varying quality.

**Fastify** — Great performance, but the plugin system has a learning curve. Schema validation is powerful but verbose.

**NestJS** — Excellent structure and DI, but heavy. The Angular-inspired patterns feel foreign to many Node.js developers.

**Hono** — Modern and fast, but young. Edge-first design can feel limiting for traditional Node.js workloads.

## What NextRush Does Differently

NextRush takes the best ideas from each framework and combines them into a cohesive experience:

| Feature | How NextRush Approaches It |
|---------|---------------------------|
| **Performance** | Radix tree routing, minimal overhead. [30,000+ RPS on typical hardware](/docs/guides/benchmarks). |
| **TypeScript** | Built from the ground up. Zero `any`. Full inference. |
| **Middleware** | Koa-style async/await with `ctx.next()`. |
| **Structure** | Optional controllers with DI when you need them. |
| **Modularity** | 27+ packages. Install only what you use. |
| **Runtimes** | Node.js, Bun, Deno, Edge — same code. |

## Core Mental Model

NextRush has three core concepts:

### 1. Application

The application is your container. It holds middleware and plugins.

```typescript
const app = createApp();
```

### 2. Context

Every request creates a fresh Context (`ctx`). This is your interface to the request and response.

```typescript
router.get('/users/:id', (ctx) => {
  const { id } = ctx.params;    // Route parameters
  const { page } = ctx.query;   // Query string
  ctx.json({ id, page });       // Send response
});
```

### 3. Middleware

Middleware transforms requests and responses. It runs in order, wrapping around your handlers.

```typescript
app.use(async (ctx) => {
  const start = Date.now();
  await ctx.next();  // Call the next middleware
  console.log(`${ctx.method} ${ctx.path} - ${Date.now() - start}ms`);
});
```

## When to Use NextRush

NextRush is a good fit when you:

- Build REST APIs or backend services
- Value type safety and want full TypeScript support
- Need high performance ([30,000+ RPS on typical hardware](/docs/guides/benchmarks))
- Want to deploy to multiple runtimes (Node, Bun, Edge)
- Prefer explicit code over hidden magic
- Like modular architecture where you install only what you need

## When Not to Use NextRush

NextRush may not be the best choice when you:

- Need a full-stack framework with SSR (use Next.js, Remix)
- Want maximum ecosystem compatibility (Express has more middleware)
- Are building a GraphQL-only API (consider dedicated GraphQL servers)
- Need a battle-tested framework for ultra-critical production (Express, Fastify are more mature)
