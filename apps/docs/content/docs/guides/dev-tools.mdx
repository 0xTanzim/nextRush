# Dev Tools

The `@nextrush/dev` package provides development and build tools that solve a critical problem: **making decorator metadata work with fast bundlers**.

## The Problem

TypeScript decorators with dependency injection require `emitDecoratorMetadata` to work. Without it, the DI container cannot resolve constructor parameters:

```typescript
@Service()
class UserService {
  // DI needs to know 'db' is of type 'Database'
  constructor(private db: Database) {}
}
```

Most modern bundlers strip this metadata for speed:

| Tool | Speed | Decorator Metadata | DI Works? |
|------|-------|-------------------|-----------|
| tsup / esbuild | ⚡ Fast | ❌ Not emitted | ❌ No |
| tsx | ⚡ Fast | ❌ Not emitted | ❌ No |
| node --strip-types | ⚡ Fast | ❌ Not emitted | ❌ No |
| tsc | 🐢 Slow | ✅ Emitted | ✅ Yes |
| **nextrush dev** | ⚡ Fast | ✅ Emitted | ✅ Yes |
| **nextrush build** | ⚡ Fast | ✅ Emitted | ✅ Yes |

NextRush dev tools give you **speed AND decorator metadata**.

## Installation

```bash
pnpm add -D @nextrush/dev
```

## Quick Start

```bash
# Start development server (auto-detects everything)
npx nextrush dev

# Build for production
npx nextrush build
```

That's it. No configuration needed. The CLI auto-detects:
- Entry file (`src/index.ts`, `src/main.ts`, etc.)
- Runtime (Node.js, Bun, or Deno)
- TypeScript settings from `tsconfig.json`

## Development Server

The `nextrush dev` command starts a development server with hot reload and decorator support.

### Basic Usage

```bash
# Auto-detects entry file
npx nextrush dev

# Specify entry file
npx nextrush dev ./src/server.ts

# Custom port
npx nextrush dev --port 4000

# Enable Node.js debugger
npx nextrush dev --inspect
```

### Options

| Option | Alias | Default | Description |
|--------|-------|---------|-------------|
| `--port` | `-p` | `3000` | Port number |
| `--watch` | `-w` | `src` | Paths to watch (repeatable) |
| `--inspect` | - | `false` | Enable Node.js inspector |
| `--inspect-port` | - | `9229` | Inspector port |
| `--no-clear` | - | - | Don't clear screen on start |
| `--verbose` | `-v` | `false` | Verbose output |

### What Happens Under the Hood

<Mermaid chart={`
flowchart LR
    A[nextrush dev] --> B{Detect Runtime}
    B --> C[Node.js]
    B --> D[Bun]
    B --> E[Deno]
    C --> F["node --watch + tsx"]
    D --> G["bun --watch"]
    E --> H["deno run --watch"]
    F --> I[Your App]
    G --> I
    H --> I
`} />

**Node.js**: Uses `node --watch` with `tsx` or `@swc-node/register` for TypeScript compilation.

**Bun**: Uses native `bun --watch` - no additional tools needed.

**Deno**: Uses native `deno run --watch` with npm compatibility mode.

## Production Build

The `nextrush build` command compiles your application with SWC, properly emitting decorator metadata.

### Basic Usage

```bash
# Build with defaults
npx nextrush build

# Custom output directory
npx nextrush build --outDir dist

# Minify output
npx nextrush build --minify

# Target ES2020
npx nextrush build --target es2020
```

### Options

| Option | Alias | Default | Description |
|--------|-------|---------|-------------|
| `--outDir` | `-o` | `dist` | Output directory |
| `--target` | `-t` | `es2022` | Target ES version |
| `--sourcemap` | - | `true` | Generate sourcemaps |
| `--no-sourcemap` | - | - | Disable sourcemaps |
| `--minify` | `-m` | `false` | Minify output |
| `--no-decorator-metadata` | - | - | Skip decorator metadata |
| `--no-clean` | - | - | Don't clean output directory |
| `--verbose` | `-v` | `false` | Verbose output |

### What the Build Produces

```
your-project/
├── src/
│   ├── index.ts
│   └── controllers/
│       └── user.controller.ts
├── dist/                     # Build output
│   ├── index.js              # Compiled JavaScript
│   ├── index.js.map          # Sourcemap
│   ├── index.d.ts            # Type declarations
│   └── controllers/
│       ├── user.controller.js
│       ├── user.controller.js.map
│       └── user.controller.d.ts
└── package.json
```

## Package.json Scripts

Add these scripts to your `package.json`:

```json
{
  "scripts": {
    "dev": "nextrush dev",
    "build": "nextrush build",
    "start": "node dist/index.js"
  }
}
```

## Multi-Runtime Support

NextRush dev tools work across all major JavaScript runtimes:

| Runtime | Dev Server | Production Build | Decorator Metadata |
|---------|------------|------------------|-------------------|
| **Node.js** | ✅ `--watch` + tsx | ✅ @swc/core | ✅ Full support |
| **Bun** | ✅ Native `--watch` | ✅ Native bundler | ✅ Full support |
| **Deno** | ✅ Native `--watch` | ✅ npm:@swc/core | ✅ Full support |

### Node.js

```bash
# Development
npx nextrush dev

# Production
npx nextrush build
node dist/index.js
```

### Bun

```bash
# Development
bun nextrush dev

# Production
bun nextrush build
bun dist/index.js
```

### Deno

```bash
# Development
deno run -A npm:@nextrush/dev dev

# Production
deno run -A npm:@nextrush/dev build
deno run --allow-all dist/index.js
```

## Entry File Detection

The CLI automatically finds your entry file in this order:

1. `package.json` `main` field (converts `dist/` to `src/`)
2. `src/index.ts`
3. `src/main.ts`
4. `src/app.ts`
5. `src/server.ts`
6. `index.ts`

## tsconfig.json Requirements

Your `tsconfig.json` must include:

```json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

<Callout type="warn" title="Required for DI">
Without these options in tsconfig.json, TypeScript won't recognize decorator syntax and the DI container won't work.
</Callout>

## Programmatic API

For build tool integration, monorepo setups, or testing frameworks:

### Development Server

```typescript
import { dev } from '@nextrush/dev';

// Simple - auto-detect entry
await dev();

// With options
await dev('./src/app.ts', {
  port: 4000,
  inspect: true,
  watch: ['./src', './config'],
  env: { DATABASE_URL: 'postgres://...' },
});
```

### Production Build

```typescript
import { build } from '@nextrush/dev';

// Simple
await build();

// With options
await build('./src/index.ts', {
  outDir: 'dist',
  minify: true,
  sourcemap: true,
  target: 'es2022',
});
```

### Runtime Detection

```typescript
import { detectRuntime, getRuntimeInfo } from '@nextrush/dev';

const runtime = detectRuntime();
// Returns: 'node' | 'bun' | 'deno'

const info = getRuntimeInfo();
// {
//   runtime: 'node',
//   version: '22.0.0',
//   supportsTypeScript: false,
//   supportsWatch: true,
//   needsSwc: true
// }
```

## Why Not Just Use tsup/esbuild?

esbuild and tools built on it (tsup, tsx) don't emit decorator metadata:

```typescript
// Your code
@Service()
class UserService {
  constructor(private db: Database) {}
}

// After tsup/esbuild (metadata LOST):
let UserService = class { constructor(db) {} };
// No Reflect.metadata! DI fails with "TypeInfo not known"

// After nextrush build (metadata PRESERVED):
let UserService = class { constructor(db) {} };
Reflect.defineMetadata("design:paramtypes", [Database], UserService);
// ✅ DI can resolve Database
```

The `nextrush build` command uses SWC with `decoratorMetadata: true`, which properly emits the reflection metadata required by tsyringe and other DI containers.

## Debugging with Inspector

Enable the Node.js debugger for step-through debugging:

```bash
npx nextrush dev --inspect
```

Then in VS Code, create a launch configuration:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "node",
      "request": "attach",
      "name": "Attach to NextRush",
      "port": 9229,
      "restart": true
    }
  ]
}
```

## Common Issues

### "TypeInfo not known for X"

This error means decorator metadata wasn't emitted. Check:
1. `tsconfig.json` has `emitDecoratorMetadata: true`
2. You're using `nextrush dev` or `nextrush build` (not tsx/esbuild directly)
3. `reflect-metadata` is imported at the top of your entry file

### "Cannot find entry file"

The CLI couldn't auto-detect your entry file. Either:
- Add a `main` field to package.json
- Create `src/index.ts`
- Specify the entry explicitly: `npx nextrush dev ./src/app.ts`

### Watch Not Working

On some systems, file watching may hit limits:
```bash
# Linux: increase inotify watches
echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```
